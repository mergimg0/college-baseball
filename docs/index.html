How It Works

College Baseball Predictions uses a three-layer prediction model to generate calibrated win probabilities for every NCAA D1 baseball game.

Layer 1: Pythagenport Expected Win% with variable exponent calibrated for college baseball (2.13, not MLB's 1.83).

Layer 2: Dynamic Bradley-Terry ratings via MM algorithm with temporal decay.

Layer 3: Log5 matchup probability with home-field adjustment (~53% baseline).

No neural networks. The accuracy comes from correct modeling of college baseball's unique run environment, not from complexity.

Frequently Asked Questions

1. Where does the data actually come from?
Three sources, all structured JSON APIs — no screen scraping or fragile HTML parsing. PEAR Ratings (pearatings.com) gives us the baseline: team ratings, power rankings, ELO, strength of schedule, and full season stats for all 294 D1 teams. NCAA Scoreboard gives us every game score, every day, across the entire D1 season. The Odds API gives us live Vegas lines — moneyline, run line, and over/unders from 40+ sportsbooks. No team websites are scraped. Everything comes through clean REST APIs that return JSON.
2. How is the data collected? What tools?
All three sources are proper APIs that return structured JSON — we just make HTTP requests and parse the response. No Selenium, no browser automation, no BeautifulSoup. It's Python's built-in HTTP library making GET requests. This keeps the system simple and reliable: no headless browsers to crash, no HTML layouts to break. If the API returns data, we have data.
3. Is the scraping logic the same for each source?
Same core pattern everywhere: build a URL, make an HTTP request, parse the JSON. The differences are minor. PEAR needs us to call once per team when we want detailed stats (about 3 minutes for all 294 teams at a polite rate). NCAA is one call per day. The Odds API is a single call that returns everything at once. All three have built-in rate limiting so we're good API citizens.
4. How often does the data refresh?
Once a day. An automated pipeline runs every afternoon: pull the latest PEAR ratings, import yesterday's and today's NCAA scores, recompute all our ratings (Pythagorean, Bradley-Terry, strength of schedule), generate tonight's predictions, render this page, and push it live. The whole pipeline takes under 5 minutes. For odds, those are fetched on-demand since lines move throughout the day.
5. How deep does the data go — team level? Player level? Play by play?
Three levels deep. Team level: 308 D1 teams with runs scored/allowed, win-loss records, conference, and a full rating suite (Pythagorean, Bradley-Terry, ELO, SOS, composite power). Player level: 9,140 pitchers with game-by-game IP/K/BB/ERA, 4,808 with advanced metrics (FIP, xFIP, SIERA from 64 Analytics), 5,130 hitters with wRCE/wOBA/ISO, and 1,113 pitchers with computed quality ratings used in predictions. Play-by-play: 245,000+ events parsed from NCAA box scores covering 18 event types (strikeouts, groundouts, hits, walks, HBP, errors, steals, etc.) with inning-by-inning scoring.
6. What happens when team names don't match across sources?
This is a real problem — NCAA might say "Mississippi St." while PEAR says "Mississippi State" and The Odds API says "Mississippi State Bulldogs." We handle it with a multi-layer system. First, exact name match. Then alias table lookup (351 known aliases covering 170+ teams). Then fuzzy substitutions (State/St., Southern/So., etc.). Then home/away swap detection for games where PEAR and NCAA disagree on who's home. When a fuzzy match works, we permanently cache it. Box score coverage is 99.9% across the season (4,145 of 4,149 games) with only 4 genuinely unavailable games.
7. What's the tech stack?
Python, all the way through. The CLI uses Click, the math uses NumPy (Bradley-Terry fitting, matrix operations), the page rendering uses Jinja2 templates, and the database is SQLite. No machine learning frameworks — our prediction model is a hand-tuned blend of Pythagorean expectation and Bradley-Terry ratings. The accuracy comes from domain-specific modeling of college baseball's run environment, not from throwing data at a neural network.
8. Where does this run? Cloud? Local?
Currently runs locally with a daily cron job, and this page is deployed as static HTML on GitHub Pages. No cloud infrastructure, no servers to maintain, no monthly hosting bill. The daily pipeline runs on a Mac, generates an HTML file, and pushes it to GitHub. For our current scale (one page, updated once daily), this is the right architecture. If we add real-time features or an API for others to query, we'd move to cloud hosting — but we're not there yet.
9. Where is the data stored?
SQLite database, about 730 KB. It holds 294 teams with full ratings, nearly 5,000 game results with scores (going back to February 14th, the start of the 2026 season), predictions, and accuracy tracking. Raw API responses are also cached as JSON files so we have a backup. SQLite is fast, zero-configuration, and perfect for this scale — the entire season's data fits in a file smaller than a single photo.
10. Can you get Vegas odds?
Yes, and it's already built. The Odds API provides moneyline, run lines (spreads), and over/unders for college baseball from 40+ sportsbooks including DraftKings, FanDuel, and BetMGM. Free tier gives us 500 API credits per month, which is more than enough for daily polling. The next step is building a betting edge calculator that compares our win probabilities against the implied odds to find value — games where our model disagrees with the market.
11. Can you backtest? Do you have historical performance?
Yes — and we take it seriously. Our backtest engine replays the entire season chronologically: for each game, it computes a prediction using only the data that was available before that game happened. No peeking at future results. It then compares our prediction against the actual outcome and against PEAR's ratings. We track accuracy (% of games called correctly), Brier score (how well-calibrated our probabilities are), and we break it down into calibration bins so we can see if "our 70%" really wins 70% of the time. The system is designed so you can never accidentally cheat — accumulators update after each prediction, not before.
